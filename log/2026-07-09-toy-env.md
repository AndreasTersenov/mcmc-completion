## Setup / what was done
- **`~/ics-env`** (fresh venv, module python/3.11.5): jax 0.9.1(+cuda12 plugin
  for GPU nodes), flax 0.12.6, optax 0.2.8, equinox 0.13.0, blackjax 1.2.4,
  numpy/scipy/pandas/matplotlib/pytest. All `pip install --no-index`.
- **Stack deviations (forced by wheelhouse incompatibilities, not choices):**
  1. **distrax dropped.** distrax 0.1.5 requires tensorflow-probability, whose
     newest wheel (0.25.0) uses `jax.interpreters.xla.pytype_aval_mappings`,
     removed in jax ≥ 0.9 → unimportable. Zoo families will be implemented
     directly in JAX (log-density + exact sampler + explicit log-det for
     warps). This aligns with the mandatory Jacobian backpressure test anyway.
  2. **diffrax de-hot-pathed.** Wheelhouse diffrax 0.5.0 → optimistix uses
     removed `jax.core.Jaxpr` → unimportable. `jax_flows` imported diffrax at
     module top; patched on a LOCAL branch `rorqual-compat` (commit in the
     clone, not pushed upstream): lazy diffrax import + scan-based
     euler/heun fixed-step samplers with identical API. FM sampling will use
     fixed-step heun (standard for FM; deterministic, jittable).
- **jax_flows audit** (user flagged "fast toy, verify"): CFM math is correct —
  OT interpolation, loss target (x1−x0), ODE direction and `sol.ys[0]` all
  check out. Two findings: (a) `sinusoidal_time_embedding` uses the
  integer-position transformer convention with t ∈ [0,1]: most channels are
  ~constant, the fastest spans <1 rad — functional but low-resolution time
  conditioning. Our own conditional nets scale t (×1000-style) before
  embedding; the reused unconditional path keeps jax_flows behavior and the
  overfit gate will catch it if it matters. (b) `train_loop` is a
  python-loop/tqdm toy — fine for gates, replaced by a scan/jit loop for real
  training. End-to-end correctness of cfm_loss+cfm_sample is enforced by a
  new analytic test (Gaussian target, closed-form OT velocity field) plus the
  overfit gate test.
- **Login-node crash root cause** (also retro-explains yesterday's fork
  exhaustion): user cgroup `pids.max = 512` on rorqual login; jax's XLA CPU
  client spawns ~192 threads (one per core) → abort when slots are scarce. A
  stale 192-thread python from an earlier import was holding 40% of the
  budget (killed). Discipline going forward, baked into the Stop hook: all
  login-node python runs under `taskset -c 0-7` with `JAX_PLATFORMS=cpu`.
- Stop hook now prefers `~/ics-env` (falls back to wl-challenge-env);
  **stage-0 suite passes unchanged in ics-env: 38/38** (backpressure (d)).
